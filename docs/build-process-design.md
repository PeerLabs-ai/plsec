# Build Process Design: bootstrap.sh

## Summary

- Makefile is the right interface: `make`, `make test`, `make clean`, `make golden`
- Templates stored as natural files (valid markdown, JSON, YAML, bash) -- independently lintable and reviewable
- A small shell assembler script reads a skeleton and substitutes template
  content with correct heredoc escaping
- Two escaping contexts: "content templates" (CLAUDE.md, opencode.json) where
  `${PLSEC_DIR}` resolves at bootstrap runtime, and "script templates" (wrapper
  scripts) where `${PLSEC_DIR}` bakes in at generation time but runtime
  variables like `$LOG_FILE` are preserved
- Build output is `bin/bootstrap.sh` -- a single distributable file, checked into version control

## Why Makefile

| Approach          | Pros                                                           | Cons                                                      |
|-------------------|----------------------------------------------------------------|-----------------------------------------------------------|
| **Makefile**      | Universal, dependency-aware, parallel targets, zero extra deps | Syntax quirks (tabs, escaping)                            |
| Just(file)        | Cleaner syntax, cross-platform                                 | Extra dependency, less universal                          |
| Python script     | Full language, Jinja2 templating                               | Wrong language for a bash build; adds Python as build dep |
| Shell script only | No extra deps                                                  | No dependency tracking, reruns everything                 |
| Task (Go)         | Fast, typed                                                    | Overkill, Go dependency                                   |

Makefile wins on universality and dependency tracking. The project already uses
bash and standard Unix tools. Make's dependency graph ensures templates are
re-assembled only when source files change.


## Template Categories

bootstrap.sh contains two distinct kinds of embedded content requiring
different escaping strategies during assembly:

### Category 1: Content Templates

Files written to `${PLSEC_DIR}/configs/` at bootstrap time. They contain
`${PLSEC_DIR}` references that resolve at bootstrap runtime.

| Template                      | Destination                            | Variables      |
|-------------------------------|----------------------------------------|----------------|
| `claude-md-strict.md`         | `${PLSEC_DIR}/configs/CLAUDE.md`       | `${PLSEC_DIR}` |
| `claude-md-balanced.md`       | `${PLSEC_DIR}/configs/CLAUDE.md`       | `${PLSEC_DIR}` |
| `opencode-json-strict.json`   | `${PLSEC_DIR}/configs/opencode.json`   | none           |
| `opencode-json-balanced.json` | `${PLSEC_DIR}/configs/opencode.json`   | none           |
| `trivy-secret.yaml`           | `${PLSEC_DIR}/trivy/trivy-secret.yaml` | none           |
| `trivy.yaml`                  | `${PLSEC_DIR}/trivy/trivy.yaml`        | none           |

**Escaping rule:** These are stored in bash variables (double-quoted strings or
heredocs). Any `$` in the template content not intended as a bootstrap variable
must be escaped. In practice, only CLAUDE.md balanced uses `${PLSEC_DIR}` --
the JSON and YAML files contain no shell variables.

**Assembly approach:** Content templates are assigned to bash variables using
quoted heredocs (`<< 'TEMPLATE_EOF'`) so nothing is interpolated during
assignment. The bootstrap logic then uses `write_file_from_var` to write them,
and `${PLSEC_DIR}` references in the content are handled via `envsubst` or
`sed` at write time.

Wait -- this changes the current architecture. Right now CLAUDE.md balanced
uses a double-quoted string so `${PLSEC_DIR}` resolves when the variable is
assigned. If we switch to quoted heredocs, we need a separate interpolation
step.

**Revised approach:** Content templates use a placeholder marker `@@PLSEC_DIR@@`
instead of `${PLSEC_DIR}`. The assembled bootstrap.sh stores them in quoted
heredocs (no escaping needed). At runtime, a `sed` pass replaces `@@PLSEC_DIR@@`
with the actual `${PLSEC_DIR}` value before writing.

This is cleaner:
- Templates contain no shell syntax at all
- They are valid, lintable files in their native format
- The substitution is explicit and auditable
- Adding new placeholders is trivial


### Category 2: Script Templates

Bash scripts generated by bootstrap and written to `${PLSEC_DIR}/`. These
contain runtime shell syntax (`$LOG_FILE`, `$(date)`, `$$`, `$@`) that must
be preserved literally in the generated file, plus `${PLSEC_DIR}` which
should bake in the actual path at generation time.

| Template              | Destination                        |
|-----------------------|------------------------------------|
| `wrapper-claude.sh`   | `${PLSEC_DIR}/claude-wrapper.sh`   |
| `wrapper-opencode.sh` | `${PLSEC_DIR}/opencode-wrapper.sh` |
| `wrapper-scan.sh`     | `${PLSEC_DIR}/scan.sh`             |
| `hook-pre-commit.sh`  | `${PLSEC_DIR}/configs/pre-commit`  |
| `pipelock-start.sh`   | `${PLSEC_DIR}/pipelock-start.sh`   |

**Current approach (in assembled bootstrap.sh):** Unquoted heredocs with
manually escaped runtime variables (`\$LOG_FILE`, `\$(date)`, etc.).
`${PLSEC_DIR}` is unescaped so it resolves from the bootstrap context.

**Template storage:** Script templates are stored as normal bash scripts with
one convention:
- `@@PLSEC_DIR@@` marks where the path should be baked in
- All other shell syntax (`$LOG_FILE`, `$(date)`, etc.) is written naturally

**Assembly transformation:** The assembler must:
1. Escape all `$` characters to `\$`
2. Escape all backticks to `` \` ``
3. Replace `@@PLSEC_DIR@@` with `${PLSEC_DIR}` (unescaped, resolves at runtime)
4. Wrap in an unquoted heredoc

This means the template files are valid, runnable bash scripts (with
`@@PLSEC_DIR@@` as the only non-standard element). They can be syntax-checked
individually by substituting a test path for the marker.


## Placeholder Convention

| Marker              | Resolves at       | Meaning                      |
|---------------------|-------------------|------------------------------|
| `@@PLSEC_DIR@@`     | Bootstrap runtime | The PLSEC_DIR path           |
| `@@PLSEC_VERSION@@` | Build time        | Version string from Makefile |
| (future markers)    | As defined        | Follow `@@NAME@@` pattern    |

Markers use `@@` delimiters to avoid collision with shell syntax, markdown,
JSON, or YAML content.

## The Skeleton

`templates/bootstrap/skeleton.bash` contains all the bash logic with
placeholder markers where template content should be inserted:

```bash
#!/bin/bash
# bootstrap.sh - Immediate security setup for AI coding assistants
# ...
# GENERATED FILE - do not edit directly
# Edit templates in templates/bootstrap/ and run 'make'

set -euo pipefail

PLSEC_DIR="${PLSEC_DIR:-${HOME}/.peerlabs/plsec}"
PLSEC_VERSION="@@PLSEC_VERSION@@"
# ... rest of config vars ...

# ... function definitions (log_*, helpers, detect_os, check_command) ...

main() {

# ... argument parsing, dependency checking, directory creation ...

# --- CLAUDE.md ---
if [[ "$STRICT_MODE" == true ]]; then
    CLAUDE_MD_CONTENT='@@INCLUDE:claude-md-strict.md@@'
else
    CLAUDE_MD_CONTENT='@@INCLUDE:claude-md-balanced.md@@'
fi

# ... write CLAUDE.md with PLSEC_DIR substitution ...

# --- opencode.json ---
if [[ "$STRICT_MODE" == true ]]; then
    OPENCODE_JSON_CONTENT='@@INCLUDE:opencode-json-strict.json@@'
else
    OPENCODE_JSON_CONTENT='@@INCLUDE:opencode-json-balanced.json@@'
fi

# ... rest of orchestration ...

# --- Wrapper scripts ---
write_file "${PLSEC_DIR}/claude-wrapper.sh" << EOF
@@INCLUDE_SCRIPT:wrapper-claude.sh@@
EOF

# ... etc ...

} # end main

if [[ "${BASH_SOURCE[0]}" == "${0}" ]]; then
    main "$@"
fi
```


## The Assembler

`scripts/assemble-bootstrap.sh` performs three operations:

1. **Read skeleton** from `templates/bootstrap/skeleton.bash`
2. **Process `@@INCLUDE:filename@@` markers** (content templates):
   - Read the template file
   - Escape single quotes (`'` to `'\''`) for safe embedding in single-quoted strings
   - Replace the marker with the file content
3. **Process `@@INCLUDE_SCRIPT:filename@@` markers** (script templates):
   - Read the template file
   - Escape all `$` to `\$` and backticks to `` \` ``
   - Replace `@@PLSEC_DIR@@` with `${PLSEC_DIR}` (unescaped)
   - Replace the marker with the transformed content
4. **Replace `@@PLSEC_VERSION@@`** with the version from Makefile
5. **Write output** to `bin/bootstrap.sh`

The assembler is ~60-80 lines of bash. It uses `sed` and `awk` for
transformations. No external dependencies.

```bash
#!/bin/bash
# assemble-bootstrap.sh - Build bin/bootstrap.sh from templates
set -euo pipefail

TEMPLATE_DIR="templates/bootstrap"
SKELETON="${TEMPLATE_DIR}/skeleton.bash"
OUTPUT="bin/bootstrap.sh"
VERSION="${1:?Usage: assemble-bootstrap.sh VERSION}"

# Read skeleton
content=$(<"${SKELETON}")

# Process @@INCLUDE:filename@@ markers (content templates)
# These go into single-quoted strings; escape single quotes
while [[ "$content" =~ @@INCLUDE:([^@]+)@@ ]]; do
    filename="${BASH_REMATCH[1]}"
    filepath="${TEMPLATE_DIR}/${filename}"
    if [[ ! -f "$filepath" ]]; then
        echo "ERROR: Template not found: ${filepath}" >&2
        exit 1
    fi
    # Escape single quotes for embedding in '...'
    template_content=$(<"$filepath")
    escaped=$(echo "$template_content" | sed "s/'/'\\\\''/g")
    content="${content/@@INCLUDE:${filename}@@/$escaped}"
done

# Process @@INCLUDE_SCRIPT:filename@@ markers (script templates)
# These go into unquoted heredocs; escape $ and ` but leave @@PLSEC_DIR@@
while [[ "$content" =~ @@INCLUDE_SCRIPT:([^@]+)@@ ]]; do
    filename="${BASH_REMATCH[1]}"
    filepath="${TEMPLATE_DIR}/${filename}"
    if [[ ! -f "$filepath" ]]; then
        echo "ERROR: Template not found: ${filepath}" >&2
        exit 1
    fi
    template_content=$(<"$filepath")
    # Escape $ and ` for heredoc, then restore @@PLSEC_DIR@@ as ${PLSEC_DIR}
    escaped=$(echo "$template_content" \
        | sed 's/\$/\\$/g' \
        | sed 's/`/\\`/g' \
        | sed 's/@@PLSEC_DIR@@/${PLSEC_DIR}/g')
    content="${content/@@INCLUDE_SCRIPT:${filename}@@/$escaped}"
done

# Replace build-time markers
content="${content//@@PLSEC_VERSION@@/${VERSION}}"

# Write output
echo "$content" > "${OUTPUT}"
chmod +x "${OUTPUT}"
echo "Built ${OUTPUT} (version ${VERSION})"
```


## Makefile

```makefile
# Makefile - plsec build and test targets

VERSION ?= $(shell grep '^version' pyproject.toml | head -1 | cut -d'"' -f2)

TEMPLATE_DIR  := templates/bootstrap
SKELETON      := $(TEMPLATE_DIR)/skeleton.bash
TEMPLATES     := $(wildcard $(TEMPLATE_DIR)/*.md) \
                 $(wildcard $(TEMPLATE_DIR)/*.json) \
                 $(wildcard $(TEMPLATE_DIR)/*.yaml) \
                 $(wildcard $(TEMPLATE_DIR)/*.sh)
BOOTSTRAP     := bin/bootstrap.sh

# --- Build ---

.PHONY: all build clean

all: build

build: $(BOOTSTRAP)

$(BOOTSTRAP): $(SKELETON) $(TEMPLATES) scripts/assemble-bootstrap.sh
	@scripts/assemble-bootstrap.sh "$(VERSION)"

clean:
	rm -f $(BOOTSTRAP)

# --- Test ---

.PHONY: test test-unit test-integration test-container test-python lint

test: test-unit test-integration

test-unit:
	bats tests/bats/unit/

test-integration:
	bats tests/bats/integration/

test-container:
	tests/bats/run-in-container.sh tests/bats/integration/

test-python:
	pytest tests/ --ignore=tests/bats

lint: lint-templates lint-bootstrap

lint-templates:
	@echo "Checking JSON templates..."
	@for f in $(TEMPLATE_DIR)/*.json; do \
		python3 -m json.tool "$$f" > /dev/null || exit 1; \
		echo "  OK: $$f"; \
	done
	@echo "Checking YAML templates..."
	@for f in $(TEMPLATE_DIR)/*.yaml; do \
		python3 -c "import yaml; yaml.safe_load(open('$$f'))" || exit 1; \
		echo "  OK: $$f"; \
	done
	@echo "Checking shell templates..."
	@for f in $(TEMPLATE_DIR)/*.sh; do \
		sed 's/@@PLSEC_DIR@@/\/tmp\/test/g' "$$f" | bash -n || exit 1; \
		echo "  OK: $$f"; \
	done

lint-bootstrap: $(BOOTSTRAP)
	bash -n $(BOOTSTRAP)

# --- Golden files ---

.PHONY: golden golden-check

GOLDEN_DIR := tests/bats/golden

golden: build
	@echo "Regenerating golden files..."
	@mkdir -p $(GOLDEN_DIR)
	@cp $(TEMPLATE_DIR)/claude-md-strict.md $(GOLDEN_DIR)/
	@cp $(TEMPLATE_DIR)/claude-md-balanced.md $(GOLDEN_DIR)/
	@cp $(TEMPLATE_DIR)/opencode-json-strict.json $(GOLDEN_DIR)/
	@cp $(TEMPLATE_DIR)/opencode-json-balanced.json $(GOLDEN_DIR)/
	@cp $(TEMPLATE_DIR)/trivy-secret.yaml $(GOLDEN_DIR)/
	@echo "Golden files updated in $(GOLDEN_DIR)/"

golden-check:
	@echo "Checking golden files match templates..."
	@diff -q $(TEMPLATE_DIR)/claude-md-strict.md $(GOLDEN_DIR)/claude-md-strict.md
	@diff -q $(TEMPLATE_DIR)/claude-md-balanced.md $(GOLDEN_DIR)/claude-md-balanced.md
	@diff -q $(TEMPLATE_DIR)/opencode-json-strict.json $(GOLDEN_DIR)/opencode-json-strict.json
	@diff -q $(TEMPLATE_DIR)/opencode-json-balanced.json $(GOLDEN_DIR)/opencode-json-balanced.json
	@diff -q $(TEMPLATE_DIR)/trivy-secret.yaml $(GOLDEN_DIR)/trivy-secret.yaml
	@echo "All golden files match."

# --- Setup ---

.PHONY: setup-bats

setup-bats:
	scripts/setup-bats.sh

# --- CI convenience ---

.PHONY: ci

ci: lint build test golden-check test-python
```


## Content Template Write Path (runtime change)

The current bootstrap.sh interpolates `${PLSEC_DIR}` at variable assignment
time using double-quoted strings. With the template system, content templates
use `@@PLSEC_DIR@@` markers and are stored in single-quoted heredocs (no
interpolation at assignment).

A small runtime helper replaces `@@PLSEC_DIR@@` when writing:

```bash
# In bootstrap.sh (skeleton)
write_content_file() {
    local dest="$1"
    local content="$2"
    # Substitute @@PLSEC_DIR@@ with actual value
    local resolved="${content//@@PLSEC_DIR@@/${PLSEC_DIR}}"
    write_file_from_var "$dest" "$resolved"
}
```

This replaces the current approach of double-quoting or unquoted heredocs
for content templates. The write path is explicit and auditable.


## Checked-in Build Artifact

`bin/bootstrap.sh` is a generated file but checked into version control.
Rationale:

- Users install via `curl | bash` -- they need a single file at a stable URL
- CI can verify the checked-in version matches `make build` output
- `make golden-check` in CI catches template/artifact drift
- The `GENERATED FILE - do not edit` header discourages direct edits

A CI step can enforce this:

```yaml
- name: Verify bootstrap.sh is up to date
  run: |
    make build
    git diff --exit-code bin/bootstrap.sh
```


## Migration Path

1. Extract templates from current bootstrap.sh into `templates/bootstrap/`
2. Create skeleton.bash by replacing inline content with markers
3. Write the assembler script
4. Write the Makefile
5. Verify `make build` produces identical output to the current bootstrap.sh
6. Add `lint-templates` and `golden-check` to CI
7. Remove the manual overlay process

Step 5 is the critical validation gate. The assembled output must be
byte-identical (or functionally identical after normalizing whitespace)
to the current hand-maintained file. If it diverges, the assembler has a bug.


## Risk: Assembler Escaping Bugs

The assembler's escaping logic (step 3 of the assembly process) is the
most likely source of bugs. Specifically:

- Script templates with `$()` command substitutions need `\$()` in the output
- Script templates with `$$` (PID) need `\$\$`
- Script templates with `$?` (exit code) need `\$?`
- Content templates with single quotes need `'\''` escaping
- Nested quotes in JSON templates

Mitigation: The `lint-bootstrap` target runs `bash -n` on the assembled
output. The integration tests verify generated file content. The golden
file check catches unexpected changes. Combined, these provide reasonable
coverage of escaping correctness.

The assembler itself should have a small set of unit tests verifying
escaping behavior on known-tricky inputs.
